# Claude Code流出で露出した「AIコーディング・ハーネス」の設計要点

## Executive Summary

Superframeworksの記事によれば、Anthropicは2026年3月31日、npm上の`@anthropic-ai/claude-code` v2.1.88に`59.8 MB`のsource map fileを誤って同梱し、`512,000 lines of TypeScript`から成るClaude Codeの全コードベースを露出させた。記事の主眼は流出そのものよりも、そこで見えた実装原則にあり、Claude Codeの優位はモデル単体ではなく、`context management, parallelism, memory, permissions`を束ねる「harness」にあると整理している。

## 流出の経緯と露出規模

記事は、原因をnpmパッケージング時のsource map混入と説明している。Bun bundlerがsource mapをデフォルト生成し、それを明示的に無効化しなかったことが発端とされる。

- **エビデンス**
- `On March 31st, 2026`, Anthropicは`*.map`を`.npmignore`に追加し忘れたと記事は記述している。
- 公開npm registry上の`@anthropic-ai/claude-code` `version 2.1.88`に、内部デバッグ用の`59.8 MB JavaScript source map file`が含まれていた。
- 露出したのは`512,000 lines of TypeScript across 2,300+ original files`。
- `By 4:23 AM ET`, security researcher `Chaofan Shou (@Fried_rice on X)`が発見を周知した。
- 初日24時間で`22 million views on X`。
- Anthropicは`DMCA takedowns`を実施したが、その後に`Python`での`clean-room rewrite`が現れ、`50,000 GitHub stars in two hours`を記録したと記事は述べる。
- 記事は、これはAnthropic npm packagesでの`second time a source map exposure`であり、`a similar incident was patched in early 2025`と記している。
- `Security Note`として、`March 31st between 00:21 and 03:29 UTC`にnpm経由でClaude Codeをインストールまたは更新した場合、`malicious version of axios containing a Remote Access Trojan`を取得した可能性があるとしている。
- **結論**
- 記事の整理では、流出対象は`harness code`であり、`No customer data`, `No API keys`, `No model weights`だったため、競争上の中核はなおClaudeのモデル側に残る。
- 一方で、`second time`という記述から、記事はこれを単発事故ではなく`systemic process issue`として扱っている。
- 露出の即時的影響は、コードの消去よりも、設計パターンの複製可能性が一気に高まった点にある。
## 512,000行から見えた10の設計要素

記事は、Claude Codeを「wrapper」ではなく「deeply engineered harness」と位置づけ、流出コードから`10 architectural secrets`を抽出している。

- **エビデンス**
- `CLAUDE.md file`は`every single prompt turn`に注入され、`40,000 characters`使える。
- 並列実行は`three distinct execution models for sub-agents`で構成される。
- `Fork`
- `Teammate`
- `Worktree`
- 権限モードは`three permission modes`。
- `Bypass`
- `Allow Edits`
- `Auto`
- メモリ圧縮は`five compaction strategies`。
- `Micro compact`
- `Context collapse`
- `Session memory`
- `Full compact`
- `PTL truncation`
- 組み込みツールは`66 built-in tools`で、`concurrent tools`と`serialized tools`に分かれる。
- フックは以下を備える。
- `pre-tool use`
- `post-tool use`
- `user prompt submit`
- `session start`
- `session end`
- フックの戻り値は`exit code 0 (allow)`, `exit code 2 (block the tool call entirely)`, `anything else (warn but continue)`。
- 永続メモリは`4 types`。
- `User memories`
- `Feedback memories`
- `Project memories`
- `Reference memories`
- 設定は`5-Layer Configuration Hierarchy`。
- `Environment Variables`
- `~/.claude/settings.json`
- `.claude.json`
- `.claude/settings.json`
- `.claude/settings.local.json`
- 起動時は`7-stage initialization`。
- `Prefetch`
- `Warnings & Guards`
- `CLI Parse & Trust Gate`
- `Setup + Tool Loading`
- `Deferred Init`
- `Mode Routing`
- `Query Engine Loop`
- サブエージェントは`four specialized types`。
- `Explore Agent`
- `Plan Agent`
- `General Agent`
- `Guide Agent`
- **結論**
- 記事の解釈では、Claude Codeの性能は単一エージェントの能力ではなく、役割分化・並列化・圧縮・権限制御の組み合わせで成立している。
- `Running a single agent is the unoptimized path.`という記述に沿えば、単独実行は基準経路ではなく、設計上の主戦場は複数サブエージェント運用にある。
- `Auto mode is the sweet spot.`、`Every permission prompt you see is a sign your configuration is incomplete`という説明から、権限確認の多さは安全性ではなく設定不足の兆候として扱われている。
- `read-only operations`を並列化し、`mutating operations`を逐次化する構成は、速度と整合性を両立させるための意図的分離として示されている。
## 記事が示す運用上の実践ポイント

記事は流出コードの分析結果を、そのまま利用者向けの運用指針に落としている。焦点は、モデルの選択ではなく、プロジェクト側の記述・設定・セッション運用に置かれている。

- **エビデンス**
- `Write a proper CLAUDE.md. It's loaded every turn.`
- `Put your architecture, coding standards, file structure, and best practices in there.`
- `You get 40K characters — use them.`
- `Use parallel agents, not a single agent.`
- `Use git worktrees to prevent branch conflicts.`
- `Use /compact proactively.`
- `Don't wait for auto-compaction to strip out context you need.`
- `Configure permissions properly. Switch to Auto mode.`
- `Set up settings.json to pre-approve safe commands.`
- `Resume sessions instead of starting fresh.`
- セッションは`persistent and saved as JSONL`で、`--continue`で再開できる。
- **結論**
- 記事の実務的含意は、`CLAUDE.md`が単なる補助ファイルではなく、毎ターン注入される主要な制御面だという点にある。
- `/compact`の推奨は、コンテキスト長の制約に受動的に従うのではなく、利用者側が要約タイミングを先回りして管理すべきだという設計思想を示す。
- `settings.json`による事前承認と`Auto mode`の組み合わせは、都度確認を減らしつつ安全性を維持するための標準運用として提示されている。
- `--continue`を使わずに新規セッションを繰り返すことは、記事の整理では、蓄積済みコンテキストを捨てて毎回`starting from zero`する運用に相当する。
## オープンソースAIコーディングツールへの波及

記事は、流出の最大の影響先をAnthropic内部ではなく、外部のAI開発ツール群に見ている。特に、設計パターンの可視化が競争条件を変えると論じている。

- **エビデンス**
- `Every open-source harness — OpenCode, Aider, Continue — now has a blueprint for what works.`
- `Projects like OpenCode, Aider, and new entrants will integrate the patterns that make Claude Code effective — parallelism, smart compaction, intelligent permissions, hook systems — into their own harnesses.`
- `Expect OpenCode and new projects to ship Claude Code-level features within weeks.`
- 記事は`The gap between proprietary and open-source AI coding tools just got significantly smaller.`と述べる。
- 同時に、`Anthropic's moat is the model, not the harness.`とも記述する。
- `Plugging open-source models into the leaked harness won't produce the same results.`
- **結論**
- 記事の立場では、流出によって縮まるのはハーネス実装の差であり、モデル能力の差ではない。
- したがって、`OpenCode`, `Aider`, `Continue`のような外部ツールは、並列化・圧縮・権限制御・フックといった周辺設計を短期間で取り込みやすくなる一方、Claude Codeの体験を完全再現できるとは限らない。
- 競争軸は「どのモデルを使うか」だけでなく、「そのモデルをどう束ねるか」に移る、というのが記事全体の中心命題である。
