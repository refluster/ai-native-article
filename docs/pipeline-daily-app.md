# Daily-App Pipeline

Turning L1–L4 into a phone-first daily app with automated batches.
The manual pages (`/l2-blog`, `/l3-insight`, `/l4-publish`) still work for
one-offs; this document covers the daily loop.

## Shape

```
You (phone):  iOS/Android share sheet → L1 PWA → Notion L1
                                                │
Daily batches:                                  ▼
  09:00 JST  L2_BATCH  one-to-one: each new L1 → one L2 draft
  10:00 JST  L3_BATCH  many-to-one: random sample of recent L2s → one L3 draft
  11:00 JST  L4_BATCH  one-to-one: each new L3 draft → publish to the site
```

Each batch is idempotent: it derives "what's already done" from the target
state, not a cursor. Re-running on the same day produces no duplicates.

## Step 1 — L1 as a PWA with Share Target

Files: `public/manifest.webmanifest`, `public/sw.js`, `public/icons/*`, `index.html`, `src/main.tsx`, `src/pages/L1Register.tsx`.

- **Installable app** via `display: standalone` + icons + service worker.
- **Web Share Target (GET)** — `share_target.action` points at `/l1-register`.
  Shared URL arrives as `?url=` (Android Chrome) or inside `?text=` (iOS
  Safari). `L1Register.tsx` extracts the first `https?://…` it finds, prefills
  the input, auto-submits **once** per navigation, then `replace`s the query
  string so a pull-to-refresh doesn't double-save.
- **Service worker** caches the app shell, uses network-first for navigations
  (so new deploys land immediately when online), cache-first for static GETs.
  Bump `CACHE_VERSION` in `public/sw.js` when shell assets change.
- **Icons** generated by `scripts/generate-pwa-icons.mjs` (zero-dep, pure
  Node + zlib). Re-run to regenerate; swap the `drawIcon` rectangles to
  redesign.

### Install

- iOS 16.4+ Safari: open `/l1-register` → Share → **Add to Home Screen**. Share
  target unlocks from the iOS share sheet once installed.
- Android Chrome: **Install app** from the address-bar menu.

## Step 2 — L1 Stats Widget

Files: `gas/src/Code.gs` (`handleL1List` now returns `createdAt` =
`p.created_time`), `src/pages/L1Register.tsx` (`computeStats`).

Three numbers above the form:

- **Today** — entries whose `created_time` falls on the local calendar day.
- **Last 7d** — count over the trailing 7 days (inclusive of today).
- **Streak** — consecutive days with ≥1 entry, ending today. A "grace" rule
  keeps the streak alive for today's zero, so long as yesterday has ≥1. Miss
  two days in a row and it resets.

Computed client-side from the same `L1_LIST` response the page already
fetches — no new endpoint.

## Step 3 — L2_BATCH (one-to-one on new L1s)

Handler: `handleL2Batch` in `gas/src/Code.gs`.

1. Query L2 DB. Collect covered L1 URLs from the `Source URLs` property.
2. Query L1 DB. Sort ascending by `created_time`.
3. Pending = L1s whose `Source URL` is not in the covered set.
4. Take up to `L2_BATCH_MAX` (3) oldest pending.
5. For each, call the existing `handleL2Create` — same code path the
   `/l2-blog` page uses.

**Why derive pending from L2 instead of a last-run cursor?** No state to
corrupt: delete an L2 draft and the next run will re-create it. No clock
skew. No "what if the trigger missed a day."

Cap of 3/run × 1 run/day = up to 3 L1→L2 conversions per day. If you bank
more L1s than that, the backlog drains FIFO at 3/day.

## Step 4 — L4_BATCH (publish new L3 drafts)

Handler: `handleL4Batch`.

1. Read `public/posts/manifest.json` from GitHub. Build set of published slugs.
2. Query L3 DB. Sort ascending by `created_time`.
3. Pending = L3s whose `id.replace(/-/g, '')` is not in the slug set. (Slugs
   are Notion page IDs without dashes — see `handleL4Publish`.)
4. Take up to `L4_BATCH_MAX` (2) oldest pending.
5. Call the existing `handleL4Publish` for each.

`manifest.json` is the only source of truth for "published." No L4 Notion DB
required (the existing `L4_DB_ID` script property is still unused by default
and the `L4_LIST` endpoint remains manual-UI-only).

## Step 5 — L3_BATCH (random sample from recent L2s)

Handler: `handleL3Batch`. Knobs at the top of the function:

| Constant | Default | Meaning |
| --- | --- | --- |
| `L3_RECENT_DAYS` | 14 | Window of L2s eligible for sampling. |
| `L3_SAMPLE_SIZE` | 3 | Number of L2s fed into one L3 synthesis. |
| `L3_AVOID_REUSE_COUNT` | 10 | Most-recently-used L2 IDs excluded from the pool. |

1. Pool = L2s with `created_time` within the last 14 days.
2. If pool < sample size → skip, return `{ processed: 0, reason: … }`.
3. Exclude the last `L3_AVOID_REUSE_COUNT` L2 IDs (stored in
   `PropertiesService` under `L3_RECENTLY_USED_L2_IDS`).
4. If filtered pool < sample size → fall back to the full recent pool (prevents
   starvation).
5. Fisher-Yates shuffle, take first `L3_SAMPLE_SIZE`.
6. Call `handleL3Create` with those IDs.
7. Prepend picks to the recently-used list; trim to `L3_AVOID_REUSE_COUNT`.

**Selection rationale.** Uniform-over-14-days gives enough variety without
sampling stale L2s. Avoid-reuse of the last 10 picks keeps synthesis topics
fresh without requiring per-L2 bookkeeping. Picking 3 L2s balances signal
density against prompt cost.

Cap 1/run × 1 run/day = **1 new L3 per day**. That pace is intentional:
L3s are the "featured" output.

## Scheduling

Handler: `setupDailyTriggers()` in `gas/src/Code.gs`. `appsscript.json`
timezone is `Asia/Tokyo`, so `atHour(9)` means 09:00 JST.

```
runL2Batch  09:00 JST
runL3Batch  10:00 JST
runL4Batch  11:00 JST
```

1-hour gaps absorb the 6-minute GAS timeout comfortably and avoid concurrent
Notion/GitHub writes.

**To install:** open the GAS editor, select `setupDailyTriggers` from the
function dropdown, **Run** once. Authorize the new
`https://www.googleapis.com/auth/script.scriptapp` scope when prompted.
(That scope is declared explicitly in `appsscript.json` under
`oauthScopes` — GAS's static scope detector misses it otherwise and
skips the authorization prompt, which would cause a `Specified
permissions are not sufficient` error on first run.) Re-running is safe:
it removes existing `runL{2,3,4}Batch` triggers before creating new ones.

**To manually trigger a batch without waiting for the schedule:** POST to
the GAS web app URL with `{ "action": "L2_BATCH" }`, `L3_BATCH`, or
`L4_BATCH`. Returned shape:

```json
{
  "success": true,
  "data": {
    "processed": 3,
    "remaining": 7,
    "items": [ { "l1Id": "…", "l2Id": "…", "title": "…" } ],
    "errors": []
  }
}
```

## State Map

| Where | Purpose | Refresh cycle |
| --- | --- | --- |
| Notion L1 / L2 / L3 databases | Content | on write |
| `public/posts/manifest.json` | Source of truth for published L3s | `L4_BATCH`/`L4_PUBLISH` |
| PropertiesService `L3_RECENTLY_USED_L2_IDS` | L3 sampling memory | `L3_BATCH` |
| Script properties (`GH_TOKEN`, `NOTION_API_KEY`, `AZURE_OPENAPI_KEY`, `L2_DB_ID`) | Secrets + DB routing | manual, set once |
| Cache `ai-native-l1-v1` (service worker) | Offline shell | bump `CACHE_VERSION` on shell change |

## Deploying the GAS changes

The React side deploys via the usual `npm run build && git push` (GitHub
Pages). The GAS side is pushed via clasp:

```bash
npx clasp push                    # uploads gas/src/Code.gs + appsscript.json
# Then in the GAS editor: Deploy → Manage deployments → New version
# And (one time) Run → setupDailyTriggers
```

The client-facing deployment URL doesn't change unless you create a *new*
deployment; "New version" under the existing deployment is usually right.

## Tuning knobs quick reference

```js
// gas/src/Code.gs
const L2_BATCH_MAX          = 3;   // L1→L2 conversions per day
const L3_RECENT_DAYS        = 14;  // L3 sampling window
const L3_SAMPLE_SIZE        = 3;   // L2s per L3 synthesis
const L3_AVOID_REUSE_COUNT  = 10;  // no-reuse memory length
const L4_BATCH_MAX          = 2;   // L3→L4 publishes per day
```

If the backlog grows faster than the batch drains it, bump the `_MAX`
values — but remember the 6-minute per-trigger ceiling. At current
~90s/publish for L4 (Notion read + markdown + image gen + 3 GitHub writes),
the safe cap is ~3–4.
