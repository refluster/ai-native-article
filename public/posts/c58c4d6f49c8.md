# マーティン・ファウラーの「on the loop」論が示す、コードレビュー中心開発から評価パイプライン設計への転換

## Executive Summary

InfoQ は 2026年3月19日、Martin Fowler ブログ上の Kief Morris の議論を引きながら、AI支援ソフトウェア開発における人間の役割が「生成物を逐一レビューする人」から「AIエージェントを導き検証する仕組みを設計・維持する人」へ移る可能性を整理した。焦点は、人間が開発ループから消えることではなく、"in the loop" でも "out of the loop" でもなく、仕様・テスト・フィードバック機構を担う "on the loop" に移る点にある。

## Martin Fowler ブログが整理した 3 つの人間関与モデル

Morris は、人間とAIの関わり方を 3 つに分け、そのうちソフトウェア工学で有効になりうる中心モデルとして "on the loop" を置く。ここで人間は、AIの出力を毎回目視確認するのではなく、AIの振る舞いを方向づける仕組みを作る側に回る。

- エビデンス
- InfoQ は、Martin Fowler のブログが「developers are unlikely to move entirely "out of the loop"」と論じたと紹介している。
- Kief Morris は 3 つの関与モデルを示した。
- "in the loop": developers review each AI output
- "out of the loop": systems operate largely autonomously
- "on the loop": humans design and maintain the mechanisms that guide and validate the system’s behavior
- Morris は、ソフトウェア工学では "on the loop" が特に有用になりうるとして、developers が「building testing frameworks, constraints, and evaluation pipelines」に注力すると述べている。
- 結論
- 論点は、AIにコード生成を任せるか否かではなく、どの層に人間の責任を残すかに移っている。
- 人間の主要タスクは、成果物単位のレビューから、仕様・制約・検証機構の設計へ再配置される。
## OpenAI Codex の "agent loop" が示す開発単位の変化

InfoQ は、OpenAI の Codex 技術解説を参照し、AI支援開発の出力が単なるチャット応答ではなく、外部ツール連携を通じて実マシン上のコード変更になると説明する。これにより、レビュー対象は「1 回の応答」ではなく、「反復的なツール使用とフィードバック循環を含む実行過程」へ広がる。

- エビデンス
- InfoQ は「Organizations across the industry are experimenting with coding agents in software development workflows」と述べている。
- OpenAI は Codex の technical deep dive で、user、model、external tools の相互作用を調整する "agent loop" を説明した。
- このアーキテクチャでは、system output は「not simply a chat response but code written or modified directly on the machine」であり、しかも「iterative tool use and feedback cycles」を通じて生成される。
- 結論
- AI支援開発の単位は、補助的なコード提案から、ツール実行を伴うエージェント的変更へ移っている。
- 人間が各生成物を逐一レビューする前提は、生成量だけでなく、生成過程の複雑さの面でも維持しにくい。
## Stack Overflow の 2025 Developer Survey が示した「利用拡大」と「信頼不足」のねじれ

AIツールの採用意向は高い一方で、AI生成コードへの信頼はそれに追いついていない。InfoQ は、このギャップが、単純な導入ではなく検証コストや保守性の問題として現れていると整理している。

- エビデンス
- InfoQ は、Stack Overflow discussions and commentary が、generative AI tools の productivity gains に対し、「maintainability and technical debt」のトレードオフを指摘していると述べる。
- とくに、generated code が production systems に安全に統合される前に、「substantial review or refactoring」を要する場合があるとされる。
- Stack Overflow の「2025 Developer Survey」では、「84% of developers reported that they are using or planning to use AI tools in their development workflows」だった。
- 一方で、「significantly fewer said they trust AI-generated output」。
- さらに、多くの回答者が「debugging AI-generated code or verifying its correctness can require additional effort」と報告した。
- 結論
- 採用率の上昇は、そのまま信頼の上昇を意味していない。
- AI生成コードの価値は、生成速度だけでは決まらず、検証・デバッグ・統合作業を含む総コストで評価される。
## Datadog の "harness-first" が置く検証の重心

InfoQ は、Datadog の議論を、AI生成物の検証を人手レビューから自動化された検証基盤へ移す具体例として扱っている。ここで中心になるのは、個々のコードを読むことではなく、仕様・シミュレーション・境界付き検証・実行時テレメトリを組み合わせたハーネスの構築である。

- エビデンス
- Datadog は recent engineering post で、「human review alone does not scale for AI-generated artifacts」と論じた。
- 理由として、「particularly when agents can produce large volumes of code」とされている。
- 代替として、teams may need to invest in「automated verification pipelines that combine specifications, simulation testing, bounded verification, and runtime telemetry」。
- Datadog はこれを "harness-first" approach と呼び、「automated verification systems evaluate agent behavior through specifications, simulation testing, and runtime telemetry」と説明している。
- このモデルでは、developers は「building the harness that validates agent outputs」に注力し、「manual inspection of each generated artifact」に依存しない。
- 結論
- AIエージェント時代の品質保証は、レビュー工程の増強ではなく、検証基盤の先行整備として構想されている。
- 人間の責任は、生成物の目視確認から、検証条件と観測可能性を設計する責任へ移る。
## 「人間を外す」のではなく、ガードレールを設計・維持する

InfoQ の記事全体は、人間の不在ではなく、人間の配置転換を主題としている。複数組織の実践や議論を通じて、AI開発ツールの周辺にある testing harnesses、evaluation frameworks、observability systems が中核に置かれている。

- エビデンス
- InfoQ は、「These discussions highlight an emerging focus on the systems surrounding AI development tools」と総括している。
- その具体物として、「testing harnesses, evaluation frameworks, and observability systems」が挙げられている。
- 記事の結論部では、Morris の "on the loop" framing は、「Rather than removing humans from the development process entirely, many teams appear to be exploring how developers can design and maintain the guardrails that shape how increasingly autonomous software systems operate」と位置づけられている。
- 結論
- 人間の役割は、AIの出力を後追いで点検することではなく、AIが逸脱しにくい運用条件を先に埋め込むことにある。
- 開発組織に求められる能力は、コーディング支援ツールの利用スキルだけでなく、ガードレールとしての仕様化・評価・監視の設計能力である。
