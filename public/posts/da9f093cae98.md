# npmレジストリのsource map流出から見えたClaude Codeの設計要点：毎ターン命令読込・共有プロンプトキャッシュ・5段階権限・25超フック

## Executive Summary

StartupHub.aiの記事によると、AnthropicのClaude Codeはnpm registry内のsource map file経由でソースコードの相当部分が露出し、内部実装として「CLAUDE.md is loaded every turn」「subagents share a prompt cache」「five-level permission system」「over 25 lifecycle events」の存在が明らかになった。

開発者にとっての要点は、単なる流出事件ではなく、長文コンテキスト処理・並列化・権限制御・拡張API設計をどう実装しているかが、具体的な設計パターンとして観測できた点にある。

## 流出の範囲と観測可能になった内部要素

記事は、流出が「an npm registry source map」に起因し、「over 2,300 original files from Claude Code's codebase」が含まれていたと述べている。報道の初出として「Twitter user @chaofan_shou (via @fried_rice)」が挙げられ、開発者コミュニティでzip fileが広く共有・分析されたとされる。

- エビデンス
- 「Anthropic's Claude Code source code was leaked via an npm registry source map」
- 「The leak consists of over 2,300 original files from Claude Code's codebase」
- 流出で確認可能になった対象として、以下が列挙されている
- 「The exact prompts and agent setup used to build and guide Claude Code」
- 「How the AI handles permissions and orchestrates multiple "sub-agents."」
- 「Specific safety features and checks implemented in the system.」
- 「The methods used for managing context and long conversations.」
- 「The underlying tool system and its extensibility through hooks.」
- 結論
- 露出したのは断片的な設定ではなく、プロンプト、権限、サブエージェント制御、長文会話管理、フック拡張までを含むアーキテクチャの主要面である。
- 競合や研究者にとっては、表面的な挙動観察ではなく、実装上の選択を直接比較できる材料が与えられた構図になる。
## CLAUDE.mdを毎ターン再読込する設計

記事は、「CLAUDE.md is Loaded Every Turn」を最初の主要発見として挙げている。これは、グローバルな指示やコーディングスタイル設定をセッション開始時だけでなく、各クエリ反復ごとに再評価する設計を意味する。

- エビデンス
- 「the CLAUDE.md file, which likely contains core instructions, coding style preferences, and other global configurations, is loaded for every single query iteration」
- 「Claude Code re-reads and re-evaluates these instructions with each turn」
- この記事はこれを「a high-leverage feature」と表現している
- 結論
- 命令を初回注入で固定せず、毎ターン再読込することで、セッション中に変化するルールやプロジェクト方針を反映しやすい構造になっている。
- 開発者視点では、システムプロンプトを一回だけ与える設計よりも、外部ファイルを反復的に解決する方式の方が、挙動の一貫性と更新容易性を両立しやすいことが示唆される。
## 共有プロンプトキャッシュによるサブエージェント並列化

記事中で最も強く打ち出されているのが、サブエージェントが「prompt cache」を共有する点である。親コンテキストをbyte-identicalに複製してforkすることで、複数エージェントを同時実行しても追加コストを抑えられるという説明になっている。

- エビデンス
- 「Claude Code's subagents share a prompt cache」
- 「When Claude Code forks a subagent, it creates a byte-identical copy of the parent context」
- 「running multiple agents simultaneously to work on different parts of a codebase incurs minimal additional cost compared to running a single agent sequentially」
- 記事はこれを「free parallelism」と位置づけ、一般ユーザーは「tend to treat Claude Code as a single worker」と述べている
- 結論
- 並列化の鍵は、エージェント数そのものではなく、親子間でどれだけ同一コンテキストを再利用できるかに置かれている。
- 同一コンテキストをbyte-identicalに保つ設計は、キャッシュ共有を成立させるための前提条件として読める。
- コードベースの別領域を同時に処理させる設計では、逐次処理より並列forkが有利になり得る、という実装上の示唆が明示されている。
## click-throughではない5段階権限設定

記事は、Claude Codeの権限モデルを「Configurable Permissions, Not Click-Through」と要約している。許可要求への「yes」クリックは機能ではなく設定不備の兆候である、という説明が特徴的である。

- エビデンス
- 「a robust, five-level permission system」
- 「designed to be configured rather than simply accepted via a prompt」
- 「When Claude Code asks for permission to perform an action, clicking "yes" signifies a failure in configuration, not a feature」
- 「glob patterns in a ~/.claude/settings.json file」
- 事前設定できる対象として
- 「always allowed」
- 「never allowed」
- 「require explicit approval」
- 結論
- 権限制御は対話時の都度承認ではなく、事前にポリシー化する設計思想に立っている。
- `~/.claude/settings.json` と glob patterns による粒度の細かい制御は、ファイルパスや操作種別に応じた多層的な許可モデルを意図している。
- 開発者向けツールで安全性を高めるには、確認ダイアログを増やすより、明示的なポリシー定義へ誘導する方が中核になる、という設計判断が読み取れる。
## 5つのコンテキスト圧縮戦略と /compact の位置づけ

記事は、長い会話に対処する仕組みとして「five different strategies」を列挙している。LLMにおける「context overflow」を中心課題と位置づけ、明示的に `/compact` の利用を推奨している。

- エビデンス
- 「Claude Code employs five different strategies to compress conversation context when it becomes too long」
- 列挙されている5戦略
- 「Microcompact: Time-based clearing of old tool results.」
- 「Context Collapse: Summarizes spans of conversation.」
- 「Session Memory: Extracts key context to a file.」
- 「Full Compact: Summarizes the entire history.」
- 「PTL Truncation: Drops the oldest message groups.」
- 「context overflow is a central problem in LLMs」
- 「Proactively using the /compact command is recommended to preserve important context and prevent the model from forgetting crucial information」
- 結論
- Claude Codeは圧縮を単一手法に依存せず、削除・要約・外部化・全体圧縮・古いメッセージ群の切り捨てを使い分ける設計を取っている。
- これは、長文会話管理を受動的な内部処理ではなく、ユーザーが `/compact` を使って介入すべき運用対象として扱っていることを示す。
- コンテキスト保持の問題はモデル性能ではなく、どの情報を残し、どれを要約・退避・削除するかという制御層の問題として実装されている。
## 実質的な拡張APIとしてのフックシステム

記事は、Claude Codeの拡張性の中核を「The Hook System as the Real Extension API」と表現している。フック対象イベントの数と、起動できるアクション種別の多さが特徴として示されている。

- エビデンス
- 「a powerful hook system with over 25 lifecycle events」
- 対象イベント例
- 「PreToolUse」
- 「PostToolUse」
- 「UserPromptSubmit」
- 「session lifecycle events」
- フックから起動できる「five types of actions」
- 「running a shell command」
- 「injecting context via LLM」
- 「running a full agent verification loop」
- 「calling a webhook」
- 「running JavaScript functions」
- 外部ワークフロー統合例
- 「auto-linting before file writes」
- 「running tests after edits」
- 「injecting relevant documentation」
- 「sending Slack notifications」
- 「validating security patterns」
- 結論
- 拡張点はツール追加そのものより、ライフサイクルの各段階に介入できるイベント駆動設計に置かれている。
- shell command、LLM経由のcontext injection、webhook、JavaScript functionsまで許すことで、Claude Codeは単体アシスタントではなく、周辺開発フローのオーケストレータとして振る舞える。
- 開発者が模倣すべきポイントは、固定機能を増やすことではなく、`PreToolUse` や `PostToolUse` のような境界面を公開し、検証・通知・文脈注入を後付け可能にする設計である。
