# 長時間自律開発を支えるハーネス設計：AnthropicのPlanner／Generator／Evaluator三層構造

## Executive Summary

Anthropicは、長時間にわたる自律的なアプリ開発で性能を押し上げる鍵として、生成担当と評価担当の分離、およびPlanner／Generator／Evaluatorから成る3エージェント構成を提示した。中核にあるのは、LLMが自分の成果物を甘く評価しがちであるという前提に立ち、外部評価ループと構造化された開発手順でその弱点を補う設計である。

同社の実験では、フロントエンド設計では5〜15反復・最長4時間の評価ループ、フルスタック開発では「Solo 20 min $9」に対して「Full harness 6 hr $200」という高コスト運用を通じて、単独エージェントより高品質な成果物を得た。一方で、モデル更新により一部の足場は不要化し、Opus 4.6ではスプリント分解を外した簡素化版ハーネスへ移行している。

## なぜ単純な長時間エージェント実装では破綻するのか

長時間タスクでの失敗要因として、Anthropicは「コンテキスト劣化」と「自己評価の甘さ」を切り分けている。前者にはコンテキスト管理、後者には評価者の分離で対処する、というのが本文の基本設計である。

- **エビデンス**
- 著者は、以前のハーネスで「initializer agent」によるタスク分解と、機能単位で実装する「coding agent」、さらにセッション間の引き継ぎ用artifactを使っていたと説明している。
- それでも複雑なタスクでは、エージェントが「go off the rails over time」する問題が残った。
- 失敗モードの1つ目として、長いタスクで「context window fills」すると一貫性を失い、一部モデルでは「context anxiety」によりコンテキスト上限が近いと判断した時点で早期に仕事を畳もうとする傾向があると述べている。
- この対策として、会話履歴をその場で要約して同じエージェントを継続する「compaction」ではなく、「clearing the context window entirely and starting a fresh agent」と、状態を引き継ぐstructured handoffを組み合わせた「context resets」を採用した。
- 以前の検証では「Claude Sonnet 4.5 exhibited context anxiety strongly enough that compaction alone wasn't sufficient」とされ、context resetが必須だった。
- 失敗モードの2つ目として、エージェントは自分が作った成果物を評価させると「confidently praising the work」しやすく、特にデザインのような主観的タスクで顕著だと指摘している。
- 著者は「Separating the agent doing the work from the agent judging it proves to be a strong lever」と述べ、独立した評価者を懐疑的に調整する方が、生成者自身を自己批判的にするより扱いやすいとしている。
- **結論**
- 長時間の自律開発では、単に1つの高性能モデルを長く走らせるだけでは不十分であり、コンテキスト制御と評価機能の分離が別々の設計課題として現れる。
- 特に「自己評価」は、テスト可能なソフトウェア開発でも設計品質のような主観評価でも共通するボトルネックとして扱われている。
- ハーネスの複雑さは副作用も持つ。本文でも「orchestration complexity, token overhead, and latency」を伴うと明記されており、性能向上と運用コストの交換関係が前提になっている。
## フロントエンド設計で主観評価を採点可能にした4基準

著者はまず、自己評価の問題が最も露出しやすいフロントエンド設計で、評価基準を明示化し、GeneratorとEvaluatorに同じ採点軸を与えるループを構築した。ポイントは「美しいか」を直接問うのではなく、設計原則に沿っているかを採点対象に変換したことにある。

- **エビデンス**
- 介入なしでは、Claudeは「safe, predictable layouts that are technically functional but visually unremarkable」に寄りやすいとされる。
- 著者はGeneratorとEvaluatorの両方に、以下の4つのgrading criteriaを与えた。
- 「Design quality」
- 「Originality」
- 「Craft」
- 「Functionality」
- 各基準の説明として、たとえば「Originality」では「template layouts, library defaults, and AI-generated patterns」を減点対象とし、「purple gradients over white cards」をAI生成らしい失敗例として明示している。
- 重み付けは「Design quality and originality over craft and functionality」。理由は、Claudeはもともと「craft and functionality」では高得点を取りやすい一方、「design and originality」では平板な出力になりやすいため。
- Evaluatorは「few-shot examples with detailed score breakdowns」で校正され、著者の好みに合わせてscore driftを抑えた。
- 実装は「Claude Agent SDK」上で行い、Evaluatorには「Playwright MCP」を付与。静止画ではなく実際のページを自律的に操作し、スクリーンショットを撮って評価した。
- 1回の生成につき「5 to 15 iterations」、各サイクルは実時間を要し、「Full runs stretched up to four hours」。
- Generatorには、評価後に「refine the current direction if scores were trending well, or pivot to an entirely different aesthetic if the approach wasn't working」と判断させた。
- 実例として、「a website for a Dutch art museum」では、第9反復で暗色系の整ったランディングページを作った後、第10反復で「a 3D room with a checkered floor rendered in CSS perspective」「doorway-based navigation between gallery rooms instead of scroll or click」へと全面転換した。
- また、基準文言に「the best designs are museum quality」を含めたところ、出力の視覚傾向が特定方向へ収束したと記している。
- **結論**
- 主観的品質でも、採点基準を具体化すれば評価ループに載せられる、というのがこの節の主張である。
- ただし、評価基準は単なる判定器ではなく、生成物のスタイルそのものを方向づける。本文で言う「criteria directly shaped the character of the output」がその含意である。
- スコアは概ね改善したが、著者自身が「I regularly saw cases where I preferred a middle iteration over the last one」と述べており、反復回数の増加が単調な品質向上を保証するわけではない。
## フルスタック開発に適用したPlanner／Generator／Evaluator構成

フロントエンド設計で得た「生成と評価の分離」を、著者はフルスタック開発に拡張した。ここでは、仕様生成・実装・QAを3エージェントに分担し、スプリント契約で高レベル仕様とテスト可能な実装の間を橋渡ししている。

- **エビデンス**
- 初期の長時間コーディングハーネスでは、context resetが重要だったが、「Opus 4.5 largely removed that behavior on its own」とされ、この新ハーネスではcontext resetを外し、「one continuous session across the whole build」に切り替え、「Claude Agent SDK's automatic compaction」がコンテキスト増大を処理した。
- 3エージェントの役割は以下の通り。
- **Planner**: 「a simple 1-4 sentence prompt」を受け取り、完全なproduct specへ展開する。詳細な実装指定ではなく「product context and high level technical design」に集中し、誤った粒度の細かい指定が下流に連鎖しないようにした。さらに「find opportunities to weave AI features into the product specs」と指示。
- **Generator**: 以前の「one-feature-at-a-time」を踏襲し、スプリント単位で1機能ずつ実装。技術スタックは「React, Vite, FastAPI, and SQLite (later PostgreSQL)」。各スプリント末尾で自己評価し、QAへ引き継ぐ。gitも利用。
- **Evaluator**: 「Playwright MCP」で実行中アプリをユーザーのように操作し、「UI features, API endpoints, and database states」を検証。採点基準はフロントエンド実験を応用し、「product depth, functionality, visual design, and code quality」を評価。各criterionに「hard threshold」があり、1つでも下回ればfail。
- 各スプリント前にGeneratorとEvaluatorは「sprint contract」を交渉し、「what "done" looked like」をコード着手前に合意した。
- 通信はファイルベースで行われ、「one agent would write a file, another agent would read it and respond either within that file or with a new file」とされる。
- **結論**
- Plannerは、短いユーザープロンプトを実装可能な仕様へ増幅する役割を担っているが、実装詳細を固定しすぎないよう制約されている。これは、誤仕様の下流伝播を避けるための設計である。
- Evaluatorは、単なる最終レビューではなく、スプリント契約という明示的な「完了定義」をGeneratorと共有することで、検証可能性を先に作ってから実装を走らせる位置づけになっている。
- 3層構造の要点は、役割分離そのものよりも、仕様化・実装・検証の境界にファイルと契約を置いて、相互作用を構造化した点にある。
## レトロゲームメーカー実験：Solo 20 min $9 と Full harness 6 hr $200 の差

著者は、単独エージェントとフルハーネスを同じプロンプトで比較し、コスト差と品質差を観察している。ここでは、ハーネスの価値は見た目の派手さではなく、中心機能が実際に動くかどうかで測られている。

- **エビデンス**
- 比較対象に用いたプロンプトは次の1文。
- 「Create a 2D retro game maker with features including a level editor, sprite editor, entity behaviors, and a playable test mode.」
- 実行条件は表で明示されている。
- 「Solo 20 min $9」
- 「Full harness 6 hr $200」
- 著者は「The harness was over 20x more expensive」と記している。
- Solo版については、最初は期待通りに見えたが、実際に触ると以下の問題が出た。
- 固定高パネルで画面の大半が空く
- スプライトやエンティティ作成を先に行う必要があるがUIが導かない
- 最も重要な点として、ゲーム自体が壊れており、「My entities appeared on screen but nothing responded to input.」
- コードを読むと「the wiring between entity definitions and the game runtime was broken」
- Full harness版では、Plannerが1文のプロンプトを「a 16-feature spec spread across ten sprints」に拡張した。
- 仕様には、コア編集機能とplay modeに加え、「a sprite animation system, behavior templates, sound effects and music, an AI-assisted sprite generator and level designer, and game export with shareable links」が含まれた。
- Plannerには「our frontend design skill」へのアクセスも与えられ、そこからvisual design languageをspecに組み込んだ。
- 実際のアプリでは、キャンバスが全画面を使い、パネル配置も自然で、見た目の一貫性があった。
- なお、UI導線の弱さは残り、「you should build sprites and entities before trying to populate a level」が依然として明示されなかった。
- 最大の差として、Full harness版では「I was actually able to move my entity and play the game.」とある。
- ただし物理挙動には粗さがあり、プラットフォーム上でキャラクターがめり込む挙動や、AI生成レベルに「a large wall that I wasn’t able to jump past」が残った。
- **結論**
- この比較で示されているのは、ハーネスが「機能数を増やした」だけでなく、単独版で壊れていた中核機能を動作可能にした点である。
- 一方で、導線の悪さや常識的なレベル設計の不足は残っており、ハーネスはプロダクト判断の全領域を自動補正するものではない。
- コストは「over 20x more expensive」であり、品質向上は無料ではない。本文の比較は、性能向上をトークン費用と実行時間で買っていることを明示している。
## QAエージェントの調整と、Opus 4.6での簡素化

Anthropicは、Evaluatorを導入すれば自動的に強いQAが得られるとは述べていない。むしろ、QAプロンプトの反復調整が必要であり、かつモデル改善に応じてハーネス構成を削るべきだという立場を取っている。

- **エビデンス**
- Evaluatorの具体的検出例として、本文は以下を表で示している。
- 「Rectangle fill tool allows click-drag to fill a rectangular area with selected tile」
→ 「FAIL — Tool only places tiles at drag start/end points instead of filling the region. fillRectangle function exists but isn't triggered properly on mouseUp.」

- 「User can select and delete placed entity spawn points」
→ 「FAIL — Delete key handler at LevelEditor.tsx:892 requires both selection and selectedEntityId to be set, but clicking an entity only sets selectedEntityId . Condition should be selection || (selectedEntityId && activeLayer === 'entity') .」

- 「User can reorder animation frames via API」
→ 「FAIL — PUT /frames/reorder route defined after /{frame_id} routes. FastAPI matches 'r eorder ' as a frame_id integer and returns 422: "unable to parse string as an integer."」

- ただし著者は「Out of the box, Claude is a poor QA agent.」と明言している。
- 初期実験では、正当な不具合を見つけても「talk itself into deciding they weren't a big deal and approve the work anyway」する傾向があった。
- また、テストが表面的で「edge cases」を十分に掘らず、微妙なバグを見逃した。
- そこで、Evaluatorのログを読み、著者の判断とずれた箇所を見つけ、「update the QA's prompt」を繰り返した。「It took several rounds of this development loop」とある。
- その後、Opus 4.6の登場を受けてハーネスの簡素化を試みた。リリースブログから、Opus 4.6は「plans more carefully, sustains agentic tasks for longer, can operate more reliably in larger codebases, and has better code review and debugging skills to catch its own mistakes」と引用されている。
- 著者はまず「Removing the sprint construct」を行い、PlannerとEvaluatorは残したが、スプリント分解を廃止した。
- Evaluatorは各スプリント評価から、実行末尾の「a single pass at the end of the run」へ移された。
- この簡素化版で、次のプロンプトを用いてDAWを構築した。
- 「Build a fully featured DAW in the browser using the Web Audio API.」
- コスト内訳は次の通り。
- 「Planner 4.7 min $0.46」
- 「Build (Round 1) 2 hr 7 min $71.08」
- 「QA (Round 1) 8.8 min $3.24」
- 「Build (Round 2) 1 hr 2 min $36.89」
- 「QA (Round 2) 6.8 min $3.09」
- 「Build (Round 3) 10.9 min $5.88」
- 「QA (Round 3) 9.6 min $4.06」
- 「Total V2 Harness 3 hr 50 min $124.70」
- QAの第1ラウンドでは、次の指摘があった。
- 「This is a strong app with excellent design fidelity, solid AI agent, and good backend. The main failure point is Feature Completeness — while the app looks impressive and the AI integration works well, several core DAW features are display-only without interactive depth: clips can't be dragged/moved on the timeline, there are no instrument UI panels (synth knobs, drum pads), and no visual effect editors (EQ curves, compressor meters). These aren't edge cases — they're the core interactions that make a DAW usable, and the spec explicitly calls for them.」
- 第2ラウンドでは、さらに以下が残課題として挙げられた。
- 「Audio recording is still stub-only (button toggles but no mic capture)」
- 「Clip resize by edge drag and clip split not implemented」
- 「Effect visualizations are numeric sliders, not graphical (no EQ curve)」
- 最終的なアプリについて著者は、プロ向けDAWには遠いとしつつも、「a working arrangement view, mixer, and transport running in the browser」を備え、エージェントがテンポやキー設定、メロディ・ドラム・リバーブ追加まで自律的に行えたと記している。
- **結論**
- Evaluatorは導入しただけでは機能せず、著者の判断基準に寄せるプロンプト調整が必要だった。本文は、評価者の分離を万能策ではなく、調整可能なコンポーネントとして扱っている。
- Opus 4.6では、以前は必要だったスプリント分解が不要になり、Evaluatorの有効性も「the task sits beyond what the current model does reliably solo」かどうかに依存する、と整理されている。
- つまりEvaluatorは固定的な必須要素ではなく、モデル能力とタスク難度の境界で効く補助輪として位置づけ直されている。
