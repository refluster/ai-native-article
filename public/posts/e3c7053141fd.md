# Sierraが再設計した「AI-native interview」――コーディング面接を廃し、Plan・Build・Reviewで測る採用信号

## Executive Summary

Sierraは、`Codex` や `Claude Code` のような coding agents によってソフトウェアエンジニアリングの重心が「機械を作ること」から「機械を設計し、磨くこと」へ移ったと捉え、面接設計を全面的に見直した。従来の coding interview や algorithms interview では日常業務に対する代表性が低く、候補者の `initiative`、`ownership`、`judgment`、`product thinking` を十分に捉えられない、というのが出発点である。

新しい `AI-native onsite` は `Plan`、`Build`、`Review` の3段階で構成され、候補者が実際にAIツールを使って2時間でプロダクトを形にする。Sierraはこの形式により、採用判断を「should we hire this person?」から「where would this person thrive, and how do we support them?」へ移したとしている。

## 従来の面接設計で失われていた信号

Sierraは、従来の面接プロセスが長く機能していた一方で、現在の業務実態との乖離が顕在化したと述べている。

- **エビデンス**
- 従来の engineering interview process は、`two coding interviews plus interviews for algorithms, system design, and culture fit, followed by reference checks` だった。
- この方式は `a well-understood, scalable approach` であり、`for a long time it worked` とされる。
- しかし近時は、面接で得られる signal の多くが `typing syntax into an editor, remembering algorithm details, stitching frameworks together` のような mechanics に偏っていた。
- その結果、`The gap showed up most clearly in debriefs.` とされ、`In the absence of clear interview signals, hiring managers leaned more heavily on referrals and prior experience.` と記されている。
- **結論**
- Sierraの問題設定は、AI時代に面接が測るべき対象が「コードを書く局所技能」ではなく、実務上の意思決定や成果創出能力へ移った、という点にある。
- 特に、面接信号が弱いときに `referrals and prior experience` への依存が強まる構図を、現行プロセスの欠陥として扱っている。
## 「AI-native onsite」の3段階設計

Sierraは coding interview と algorithms interview を廃止し、実務再現性を重視した onsite に置き換えた。

- **エビデンス**
- `We removed our coding and algorithms interviews and replaced them with an AI-native onsite:` と明記している。
- 構成は以下の3段階である。
- `Plan`: 候補者とともに build する product を定義する working session。`The candidate drives ideation, while interviewers ask questions to strengthen it.` また、`We focus on an idea in the candidate’s domain so we see their product thinking in action.`
- `Build`: 面接官が退出し、候補者が `2 hours` でアイデアを形にする。`using the AI tooling and frameworks of their choice` であり、`complete freedom to pivot or adjust scope as they go` とされる。
- `Review`: 候補者が demo を行い、`the key product flows and choices they made` を議論し、コードを見ながら `data model, abstractions, extensibility, etc.` を確認し、さらに `the path to production` と `how they used AI along the way` を話す。
- この新形式は、`more representative of the work, and produces higher signal` と評価されている。
- Sierraは具体的に、`agency (do they pivot when they get stuck?)` と `judgement (how do they scope what to build within the time constraints?)` を見やすくなったと述べる。
- **結論**
- この設計は、候補者に「何を作るか」「どこを削るか」「AIをどう使うか」を実時間で選ばせることで、成果物そのものよりも意思決定過程を評価対象にしている。
- `Plan` と `Review` を含めることで、単なる vibe-coding 能力ではなく、プロダクト定義、技術判断、運用導線までを一連の職務として見る枠組みになっている。
## 候補者体験と評価基準の再定義

Sierraは、この面接を高信号であるだけでなく、候補者にとっても「engaging and authentic」なものにすることを設計要件に置いている。

- **エビデンス**
- AI-native interview process の設計要件として、以下の `three key attributes` を挙げている。
- `Representative`
- `High signal`
- `Positive experience`
- `Positive experience` については、`Feels engaging and authentic for candidates, so that when we make an offer, they’re excited to say yes.` と定義している。
- 候補者の不安に備え、`we share evaluation criteria and advice ahead of time` としている。
- 事前に伝える助言の例として、`it’s OK to cut your scope as you build, and to skip boilerplate (CRUD, auth) to focus on what’s unique.` を挙げている。
- さらに `As Paul Buchheit, the creator of Gmail, put it : if it’s great, it doesn’t have to be good.` と引用している。
- 候補者フィードバックについて、`many have said this was the most fun they’ve had in an interview.` と記載している。
- 実例として、
- `One trivia enthusiast built an AI-powered game intended to keep the user in a state of flow — the demo just involved the interviewer playing it.`
- `In another case, a backend engineer built a headless simulation tool and used an agent with a markdown file to walk through the demo.`
- **結論**
- Sierraは、評価の透明性を上げることで open-ended な面接の不安を抑え、候補者の創造性を引き出そうとしている。
- `CRUD, auth` のような boilerplate を明示的に評価対象から外し、`what’s unique` に集中させる方針は、AI時代の差分がどこにあるかを面接設計へ反映したものと読める。
## 電話面接とデバッグ面接の再構成

Onsite だけでは測れない能力に対して、Sierraは system design interview と debugging interview を追加・試行している。

- **エビデンス**
- 従来の `coding phone screen` は、`write code without AI into an online editor` を要求していた。
- しかし Sierra は、`vibe-coding an app is easy. The harder, more relevant, problem is getting it into production in a scalable way.` と述べ、これを `a system design interview` に置き換えた。
- AI-native onsite は `product sense and building 0->1` を測る一方で、`taking a feature from 1->N in an existing, messy codebase` は捉えにくいと認めている。
- その補完として、`we’re piloting a debugging interview.` としている。
- この debugging interview では、候補者に `a medium-sized codebase and a draft PR from a colleague that introduces a cross-cutting feature` を渡し、`review and improve it` することを求める。
- 候補者は `pulling down the code, inspecting the output, and iterating with coding agents to make it better.` とされる。
- ただし AI 利用の水準については、`The level of AI used in this interview is still TBD, as new models can zero-shot many fixes.` と留保している。
- **結論**
- Sierraは、0→1 の創造力と、既存コードベースでの 1→N の改善能力を別々の面接で測る構造に切り分けている。
- 特に debugging interview は、AIが `zero-shot many fixes` できる状況下で、候補者の価値を「修正そのもの」ではなく、レビュー、検証、改善の運転能力へ移している。
## 「弱みの不在」ではなく「強み」で採るという運用思想

Sierraは、新しい面接フォーマットによって採用会議の論点そのものが変わったとしている。

- **エビデンス**
- `We’re hiring for strengths, not just an absence of weakness.` と明言している。
- この方式により、`much richer signal about a candidate’s spikes and gaps` が得られるという。
- 例として、`some people excel at product strategy and initiative but have holes in their system understanding.` と述べている。
- その結果、debriefs は `“should we hire this person?”` から `“where would this person thrive, and how do we support them?”` へ変化した。
- 一方で課題として、`It’s open-ended, which makes it harder to standardize.` と認めている。
- その緩和策として、`a set of evaluation criteria that are agnostic to what the candidate builds` を整備し、`we run interviews in pairs to improve calibration.` としている。
- 適用範囲については、`We also debated whether this approach applies for infrastructure, and concluded that it does` とした上で、`we’ve amended the interview slightly to better capture the signal we need for infrastructure.` と記している。
- 全体の背景として、`The emergence of highly proficient coding agents is forcing us to reimagine Sierra from the ground up — from how we build (using agents to build and optimize agents with Ghostwriter), to how we hire.` と総括している。
- **結論**
- Sierraの採用思想は、標準化された減点型評価から、役割適合と支援設計を含む配置型評価へ移っている。
- ただし open-ended であるがゆえに、評価基準の抽象化と `pairs` による calibration を不可欠な運用装置として置いている。
